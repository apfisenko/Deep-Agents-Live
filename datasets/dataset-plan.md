## 4. Формат записи (ChatML)

### `input` — окно диалога

Массив сообщений **до точки, где агент должен ответить** (не включая ответ агента):

```json
{
  "input": [
    {"role": "user", "content": "..."},
    {"role": "assistant", "content": "..."},
    {"role": "user", "content": "..."}
  ],
  "expected_output": "...",
  "metadata": { }
}
```

- Роли: `user` = peer, `assistant` = эксперт (исторические реплики в окне)
- Точка среза: последняя реплика `user` в окне — триггер ответа агента
- Окно включает все предыдущие реплики чата (без скользящего окна — чаты короткие, ≤18 сообщений)

### `expected_output`

- Для **информационных** кейсов: эталонный текст ответа (на базе ответа эксперта + правки под агента)
- Для **tool-кейсов**: структура в `metadata.expected_tools` + краткий финальный текст

### `metadata` (обязательные поля)

| Поле | Тип | Описание |
|------|-----|----------|
| `group` | string | Группа из таксономии (1–7) |
| `category` | string | Подкатегория (intent-метка) |
| `source` | `extracted` \| `synthetic` | Происхождение |
| `chat_id` | string? | Для extracted: `CHAT_00XX` |
| `turn_index` | int? | Индекс последней user-реплики |
| `segment` | `b2c` | Всегда b2c в этом датасете |
| `ability` | string | S1–S7 из таблицы выше |

## 6. Метрики

| Тип | Форма ground truth | Метрика |
|-----|-------------------|---------|
| T1 logistics_qa | `metadata.checklist` (подвопросы) + эталонный текст | LLM-judge Answer Correctness + checklist coverage (код) |
| T2 product_fit | `metadata.expected_product`, `expected_rag_topics` | Product accuracy (код) + Answer Correctness (judge) |
| T3 format_objection | `metadata.expected_action` (waitlist / alt_cohort) | Action match (код) + judge |
| T4 trust_proof | эталонный текст + `must_not_include: [reviews_only]` | Judge с рубрикой «контент до квалификации» |
| T5 sales_friction | `metadata.qualification_framing: value_first` | Judge по рубрике |
| T6 deferral_cta | `metadata.expected_cta` (link / date / question) | CTA presence (код) + judge |
| T7 pain_feedback | `metadata.expected_rag_topics` + техответ | RAG topic hit + judge |
| T8 tool_invoke | `metadata.expected_tools[]` с name + args | Tool call accuracy (код, стадия Evaluation) |

**Retrieval-метрики** (Hit@k, MRR) — когда в `data/b2c/` появятся стабильные chunk id; пока — `expected_rag_topics` как proxy.

**Не измеряем в v0.1:** B2B-сценарии (отдельный корпус), security.

## 7. Техники

| Техника | Где применяем |
|---------|---------------|
| **Стратификация** | Синтетика: минимум 1 запись на слабую подкатегорию из матрицы пробелов |
| **SGR / structured output** | Извлечение (Уровень 2): Pydantic-схема с `category: Literal[...]` из таксономии |
| **Эталон = реальный фрагмент** | Извлечение: `reference` дословно из диалога, правки только на апруве человека |
| **Негативные инструкции** | Синтез: не генерировать B2B, не дублировать уже извлечённые формулировки |
| **Персоны** | Синтетика: ~20% baseline / ~80% перефраз под роли (CPO, разработчик, новичок) |
| **RANDOM_SEED** | Фиксировать в `metadata.seed` для воспроизводимости |
| **Окно диалога** | Все записи — multi-turn input (решение зафиксировано) |

## 8. План валидации

1. **Структурная** — JSON Schema / Pydantic: поля `input`, `expected_output`, `metadata`
2. **Покрытие** — все группы 1–7 представлены; T4 = ровно 4 записи (1+3)
3. **Семантическая** — окна не дублируют один и тот же turn; synthetic ≠ copy extracted
4. **Validation Sample** — 5 записей глазами до полного объёма (гейт после выборки)
5. **Прогон сильной моделью** — 10 записей через агента; если judge < порога — правим эталон, не агента
