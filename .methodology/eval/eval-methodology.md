# Eval-методология (AI-Driven Evaluation)

> **Философия:** качество LLM-агента — это измеренная величина, а не впечатление.
> Конфигурация без прогона — мнение. Прогон без зафиксированной конфигурации — шум.
> Человек отвечает за эталоны, критерии и решения; агент — за механику исполнения.

Методология описывает, как поставить и вести измерительный контур качества LLM-агента: от решения «строим датасеты» до сравнения конфигураций и регрессионного гейта — предсказуемо, трассируемо и повторяемо. Является расширением общей AI-Driven Methodology проекта: те же слои (roadmap → sprint → task), тот же 4-фазный цикл задач и правило двух согласований.

## Концепции: из чего мы исходим

Теоретическая основа методологии — **AI Engineering Loop** ([Langfuse Academy](https://langfuse.com/academy/ai-engineering-loop)) и методология оценки агентов [Google ADK](https://google.github.io/adk-docs/evaluate). Ниже — адаптированные выжимки с ссылками на оригиналы; принципы E-* в следующем разделе выводятся из этих концепций.

### К-1. Качество — процесс, а не состояние

LLM-приложение отличается от традиционного ПО вероятностным выводом: один и тот же промпт может вернуть разные ответы, входное пространство огромно, граничные случаи нельзя предусмотреть заранее. Следствие: одноразовый запуск тестов перед деплоем ничего не гарантирует — нужен **цикл, замыкающий продакшн и разработку**:

```
Блок 1 — понять, что происходит:        Trace → Monitor (+ Error Analysis)
Блок 2 — улучшать, не ломая работающее: Build datasets → Experiment → Evaluate
→ деплой → цикл начинается снова
```

Не обязательно стартовать со всеми шагами сразу — ценность накопительная, каждый добавленный шаг улучшает сигнал. Наша методология реализует Блок 2, опираясь на уже работающие Trace/Monitor.
*Источник: [AI Engineering Loop](https://langfuse.com/academy/ai-engineering-loop).*

### К-2. Трейс — сырьё и точка диагностики

Трейс — структурированный след каждого запроса (промпт, retrieved-контекст, вызовы инструментов, ответ, latency, стоимость) с иерархией шагов. Без трейсинга мониторить нечего, датасеты брать неоткуда, эксперименты сравнивать не с чем. Score говорит «что плохо», трейс — «где сломалось»: любой плохой результат раскручивается до конкретного шага системы (component-level диагностика).
*Источники: [Academy · ai-engineering-loop](https://langfuse.com/academy/ai-engineering-loop), [Langfuse data model](https://langfuse.com/docs/observability/data-model).*

### К-3. Провалы доменно-специфичны → error analysis раньше дизайна метрик

«RAG достаёт не тот раздел. Support-бот не замечает follow-up. Агент берёт не тот инструмент» — провалы специфичны для домена, библиотечные evaluators дают лишь отправную точку. Структурный метод **error analysis** (5 шагов): gather traces → **open coding** (свободные заметки «что первым пошло не так», без заранее заданных категорий) → **cluster** (таксономия сбоев) → **label & measure** (failure rate по категориям) → **decide & act** (разово починить / поставить evaluator / мониторить). Запускать: перед дизайном evaluators, после смены промпта/модели, при сигнале мониторинга, и регулярно — первая таксономия никогда не финальная.
*Источник: [Academy · Error Analysis](https://langfuse.com/academy/monitoring/error-analysis).*

### К-4. Датасет — мост между мониторингом и экспериментами

Датасет = коллекция тест-кейсов, против которой прогоняют приложение при каждом изменении. Анатомия item: `input` / `expected_output` / `metadata`. Паттерны `expected_output`: **exact match** (литеральный ответ), **reference answer** (эталон для семантического сравнения), **evaluation criteria** (список требований), **ничего** (reference-free метрики), **комбинация** (JSON с несколькими типами сразу). Хороший датасет имеет чёткий скоуп (один датасет — одна зона ответственности) и размер под задачу: ~10 items для расследования одной проблемы, 100–1000 для CI-проверки. Три источника items: **продакшн-трейсы** (`+ Add to dataset` с `source_trace_id`), **вручную** (требования, edge cases), **синтез** (после реальных примеров-затравок). Annotation Queues — мост для ручной разметки: corrected outputs становятся эталонами, та же разметка — human baseline для калибровки LLM-судьи.
*Источники: [Academy · Datasets](https://langfuse.com/academy/datasets), [Docs · Datasets](https://langfuse.com/docs/evaluation/experiments/datasets), [Annotation Queues](https://langfuse.com/docs/evaluation/evaluation-methods/annotation-queues).*

### К-5. Эксперимент — сравнение, а не абсолютная оценка

Вопрос эксперимента — не «хорошо ли это?», а «лучше или хуже, чем было?» при изменении **одной** переменной. Стабильный baseline — точка отсчёта; каждый прогон — dataset run с привязкой к версии датасета, scores per item и в среднем. Отсюда: главная метрика, неприкосновенный baseline, правило одного параметра, сравнение только на одной версии датасета.
*Источник: [Academy · Experiments](https://langfuse.com/academy/experiments).*

### К-6. Метрики пути и метрики исхода взаимодополняющи

Rubric/judge-метрика сама по себе может пропустить галлюцинацию, которая «хорошо читается»; trajectory-метрика сама по себе не скажет, получил ли пользователь полезный ответ. На важных кейсах — обе. Словарь строгости сравнения траекторий: EXACT / IN_ORDER / ANY_ORDER; multi-turn сценарии проверяются user simulation.
*Источник: [ADK · Evaluate](https://google.github.io/adk-docs/evaluate).*

## Слой 0: Предусловия eval-трека

**Eval-трек нельзя начинать раньше, чем продукт поставлен в воспроизводимом виде** (следствие К-1/К-2: Блок 2 цикла опирается на Блок 1).

- [ ] Агент работает end-to-end (не заглушка), доступен по API
- [ ] Трейсинг в Langfuse настроен; спаны инструментов (MCP tools, retrieval, LLM-вызовы) видны как отдельные observations
- [ ] Существует (или создаётся первой задачей) **реестр конфигураций агента**: Agent Core умеет применять конфигурацию по `config_id`
- [ ] Есть ≥ 10 кейсов с известным правильным ответом (реальные диалоги + база знаний)
- [ ] Определена **главная метрика** — что считать «лучше» при сравнении конфигураций
- [ ] Есть ≥ 2 конфигурации, которые хочется сравнить

## Слой 1: Принципы

Принципы пронумерованы; при дрейфе исполнения ссылаться на номер («сверься с E-7»).

### Архитектура контура

- **E-1. Git — источник правды.** Версии датасетов, конфигурации, промпты-указатели, карты и решения живут в репозитории: history, review, rollback. Langfuse — среда исполнения и анализа, синхронизируется автоматизированно.
- **E-2. Автоматизация и повторяемость.** Каждая операция контура (`validate / sync / experiment / analyze / compare`) оформлена как повторяемая команда — рекомендуется make как единая точка входа, допустимы иные task-runner'ы и CI. Запрещены неповторяемые ручные операции как канал изменений: правка данных в UI Langfuse не является источником правды — её результат возвращается в файлы (export → PR), иначе теряется.
- **E-3. Не форкать агента.** Бенчмарк гоняет ту же продакшн-цепочку, что работает с пользователями (Agent Core API / фабрика с `config_id`). Иначе измеряется не то, что в проде.
- **E-4. Eval — отдельный трек.** Собственный `docs/roadmap-eval.md`, пространство спринтов `docs/sprints/eval/sprint-eval-NN-<name>/`, свой темп.

### Конфигурации запуска

- **E-5. Конфиг полностью описывает систему:** impl агента, retrieval-бэкенд, модель агента, **модель judge** (`judge:` — обязательный блок, отдельно от `model:`), промпт (имя + версия/label), пины версий датасетов, комментарий. Шаблон: [`templates/run-config-template.yaml`](../templates/eval/run-config-template.yaml).
- **E-6. Конфиг исполняем, а не декларативен:** `config_id` реально применяется к агенту через реестр конфигураций.
- **E-7. Один параметр** между сравниваемыми конфигами; изменение системы = новый конфиг-файл; baseline неприкосновенен.
- **E-8. Benchmark-only конфиги:** новые модели/настройки — сначала с пометкой `benchmark_only: true`, в продукт — после прохождения бенчмарка.
- **E-9. Полная фиксация рана.** Имя: `{config_id}--{dataset}--{git_sha8}--{ISO-ts}`; в `run_metadata` — конфиг целиком, judge, резолвленная версия промпта, `git_sha`, имя и версия датасета; **тайминги** (`duration_ms` per item, `total_duration_ms` run).
- **E-27. Локальный JSON — source of truth.** Каждый прогон обязан писать `evals/reports/runs/<run_name>.json` (schema v2): полный snapshot конфига, judge, scores, items, timing. Langfuse — зеркало для UI; analyze/compare читают только локальные отчёты.

### Промпты

- **E-10. Промпты — в Langfuse Prompt Management:** правка = новая иммутабельная версия, продвижение — labels; конфиг ссылается по имени + label/версии, при запуске резолвится точный номер и пишется в metadata рана.

### Датасеты

- **E-11. Группировка и версионирование в файловой системе.** Датасеты группируются по типам (слоям оценки): `evals/datasets/<group>/<name>/`, где group ∈ {`e2e`, `rag`, `behavior`, `edge`}. Внутри папки датасета — **иммутабельные файлы-версии** `vNNN_YYYY-MM-DD.<ext>`: правка датасета = новый файл со следующим номером, старые версии не редактируются (история версий = листинг папки). Рядом — `README.md` датасета ([шаблон](../templates/eval/dataset-readme-template.md)) и `scenarios.yaml` для user-simulation датасетов.
- **E-12. Формат манифеста — человекочитаемый и diff-friendly.** YAML, JSON, JSONL или CSV — на выбор под датасет; критерий один: эталон читается в review/diff **без декодера** (битмаски и числовые коды запрещены). Рекомендация: YAML/JSON для малых ручных датасетов, JSONL/CSV — для крупных и выгрузок. Единая схема item — [`templates/dataset-manifest-template.yaml`](../templates/eval/dataset-manifest-template.yaml): `id` (стабильный ключ upsert), `input`, `expected_output` (любой паттерн из К-4), `metadata{segment, intent, source, gt_quality, reviewed_by}`.
- **E-13. Human review — гейт:** item без `reviewed_by` не проходит валидацию синка. Эталоны утверждает человек (LLM, генерирующий и эталоны, и ответы, и оценку — замкнутый круг).
- **E-14. Честность ground truth фиксируется явно:** `gt_quality: verified | approximate`. Пока есть approximate-эталоны, метрики валидны для относительного сравнения конфигов, а не для абсолютных утверждений.
- **E-15. Типизация и целостность:** манифесты грузятся через Pydantic-модели; integrity-тесты ([шаблон](../templates/eval/test-dataset-integrity-template.py)) — часть валидации. Дополнительно платформенный уровень: Langfuse Schema Enforcement (JSON Schema на input/expected_output датасета). Опечатка должна падать на валидации, а не в середине прогона.
- **E-16. Зеркалирование версий в Langfuse.** Файлы-версии — источник правды (E-1); в Langfuse версия отражается одним из двух способов, выбор фиксируется в ADR проекта: **(а) folders-as-versions** (по умолчанию): имя датасета `<group>/<name>/vNNN` — слэш создаёт виртуальные папки в UI, каждая версия — отдельный иммутабельный датасет-зеркало файла; **(б) нативное версионирование** Langfuse (timestamp time-travel, эксперименты на версии-срезе) — удобно, когда правки идут в UI, но тогда обязателен export обратно в файл (E-2). Сравнивать прогоны можно только на одной версии датасета — `compare` обязан это проверять.

### Метрики

- **E-17. Framework-first.** Выбор строго по [`metrics-guide.md`](metrics-guide.md): (а) детерминированная проверка кодом → (б) готовая метрика RAGAS/DeepEval → (в) G-Eval для своего критерия → (г) свой судья только через ADR ([шаблон](../templates/eval/adr-custom-metric-template.md)). Перед дизайном кастомных метрик — error analysis (К-3): таксономию задают трейсы, а не абстрактные критерии.
- **E-18. Главная метрика** сравнения конфигов одна (для llmstart: `avg_answer_correctness` на `e2e-qa`) + guard-метрики (faithfulness, task_completion, error_rate). «Конфиг лучше» = главная выросла при не-просевших guard.
- **E-19. Два уровня scores:** item-level (значения метрик, `task_error`) и run-level (агрегаты, `error_rate`). Упавший item — метрика, а не исключение в логе. Шаблон evaluators: [`templates/eval/evaluators-template.py`](../templates/eval/evaluators-template.py).
- **E-20. Пороги — до прогона,** меняются только решением, не подгонкой.
- **E-21. ToolCorrectness — семантика IN_ORDER** по умолчанию; EXACT — осознанным решением в metrics-map.

### Цикл улучшения

- **E-22. Eval-fix loop:** одно изменение за итерацию; регрессия на зелёных важнее улучшения на красных; норма 5–10+ итераций.
- **E-23. Multi-turn — через user simulation** (`scenarios.yaml` + детерминированный критерий успеха + TaskCompletion; весь сценарий — один трейс).
- **E-26. Каждый эксперимент фиксируется протоколом.** Машинная фиксация (E-9, run_metadata) отвечает на «что гонялось»; протокол — на «зачем и что решили». Перед запуском: строка в журнале [`evals/reports/experiments-log.md`](../templates/eval/experiments-log-template.md) (гипотеза, датасет@версия) + протокол по [шаблону](../templates/eval/experiment-report-template.md) со статусом 🚧. По завершении: результаты, находки и **решение** (принят / отклонён / итерация) — в протокол и журнал. Отрицательные результаты не удаляются — они защищают от повторения уже проверенных гипотез. Эксперимент без протокола считается не проведённым.

### Платформа

- **E-24. Langfuse как среда исполнения:** Dataset (items) → Experiment/Dataset Run (каждый прогон = новый run с metadata и агрегатами) → каждый item связан с полным trace → Score на run/trace (end-to-end) или span (component-level), reasoning судьи — в comment. Runs сравниваются side-by-side в UI.
- **E-25. Версия SDK пинится** в lockfile (Experiments — pre-GA API); **major SDK == major Docker image** (`check_langfuse_versions.py` в `make eval-validate`); run-level ключи — только в `run_metadata`, не в metadata items; длинные списки (`facts`, `must_not`) — в `expected_output`, в metadata только счётчики/превью ≤200 символов (OTEL propagation); перед апгрейдом SDK — сверка через Langfuse Docs MCP.

## Слой 3: Спринты и задачи

Полностью по общей методологии: спринты в `docs/sprints/eval/sprint-eval-NN-<name>/`, задачи с 4 фазами и двумя ⛔ СТОП-гейтами. Специфика eval-задач:

- В DoD задач прогона — **evidence**: скриншот Dataset Run + проверка drill-down (спаны агента вложены в trace item) в `summary.md`.
- Задачи разметки эталонов содержат явный шаг «эталоны показаны пользователю и утверждены» (E-13).
- DoD спринта собирается из чек-листов этого документа, а не придумывается заново.
- Планы задач — по шаблону [`templates/task-plan-eval-template.md`](../templates/eval/task-plan-eval-template.md).

**Первый спринт стандартизован:** агент инстанцирует его из канона [`templates/eval/sprint-eval-canon-template.md`](../templates/eval/sprint-eval-canon-template.md) (6 задач: реестр+каркас → **карта датасетов ⛔** → **карта метрик ⛔** → манифест+review+sync → runner+baseline → отчёт; карты — две отдельные последовательные задачи: метрики проектируются под утверждённые датасеты, не наоборот).

**Последующие спринты** порождаются из KR этапов roadmap-eval: eval-02 — остальные датасеты по dataset-map; спринты v0.2 — compare, candidate-конфиги, eval-fix loop, user simulation, error analysis; v1.0 — skill, CI-gate, annotation queues. Спринты впрок не создаются — dataset-map после первого среза почти наверняка уточнится.

## Поток работы: от «строим датасеты» до сравнения конфигураций

**Подготовка (онбординг трека).** Выполняется агентом по этому документу, без отдельных промптов:
1. Проверить предусловия Слоя 0 по факту репозитория, статусы зафиксировать.
2. Нет `docs/roadmap-eval.md` — создать из [шаблона](../templates/eval/roadmap-eval-template.md). ⛔ СТОП — ревью.
3. Нет открытого eval-спринта — создать `sprint-eval-01-vertical-slice` из [канона](../templates/eval/sprint-eval-canon-template.md). ⛔ СТОП — ревью.
4. Дальше — цикл задач по общей методологии (Plan → ⛔ → Implement → Self-check → ⛔ → Record).

> **⚠️ Граница онбординга (правило агента №8).** Онбординг производит ровно два документа планирования: `roadmap-eval.md` и `README.md` спринта. **Все рабочие артефакты** — dataset-map, metrics-map, конфиги, манифесты, скрипты — создаются **только внутри задач спринта**, после апрува `plan.md` соответствующей задачи. KR в roadmap — это обещания будущих спринтов, а не todo-список текущего шага: желание «заодно» создать карту при создании roadmap — сигнал нарушения, остановись.

Kickoff-фраза, запускающая всё вышеперечисленное:
> «Приступаем к eval-треку. Иди по .methodology/eval/eval-methodology.md, раздел "Поток работы → Подготовка": проверь предусловия, создай roadmap-eval и sprint-eval-01 из шаблонов методологии, останавливайся на каждом ⛔.»

**Цикл первого среза** (соответствие задачам канона — в Слое 3):

1. **Каркас и реестр конфигов** → структура `evals/`, команды операций, скелеты скриптов, baseline-конфиг; Agent Core принимает `config_id`. ⛔ конфиг полностью описывает систему (E-5) и исполняем (E-6).
2. **Карта датасетов** → `docs/eval/dataset-map.md` по [шаблону](../templates/eval/dataset-map-template.md): входы — концепт-дока (сценарии), реальные диалоги, отчёт анализа; группы по слоям (E-11), каждое решение с обоснованием. ⛔ человек утверждает карту датасетов.
3. **Карта метрик** → `docs/eval/metrics-map.md` по [шаблону](../templates/eval/metrics-map-template.md), строго под **утверждённый** dataset-map и по metrics-guide (E-17): главная метрика (E-18), пороги (E-20). ⛔ человек утверждает карту метрик.
4. **Манифесты** → `v001_*.yaml` + Pydantic + integrity-тесты; эталоны на утверждение; `reviewed_by`, `gt_quality`. ⛔ review эталонов (E-13). Затем sync (идемпотентность — повторным запуском).
5. **Baseline-прогон** → строка в журнале + протокол эксперимента 🚧 (E-26); item- и run-evaluators (E-19); evidence в summary. Baseline фиксируется в протоколе как точка отсчёта.
6. **Анализ** → отчёт + разговорный разбор худших items по трейсам; при системных провалах — error analysis по К-3 (open coding → таксономия → измерение), категории сбоев пополняют датасеты (цикл К-4). Находки — в протокол эксперимента.
7. **Eval-fix loop** → исправление (новый конфиг при изменении системы) → прогон → compare → дельты и регрессии (E-22). Каждая итерация = эксперимент с протоколом и решением в журнале (E-26).
8. **Институционализация** → skill, CI-gate, annotation queues, расширение разметки approximate → verified.

## Чек-листы

### При постановке контура
- [ ] Реестр конфигураций работает: `config_id` реально меняет поведение агента (E-6)
- [ ] Манифесты читаемы без декодера (E-12); Pydantic + integrity-тесты зелёные (E-15)
- [ ] Sync идемпотентен (повторный запуск — 0 новых items)
- [ ] Способ зеркалирования версий в Langfuse выбран и зафиксирован (E-16)
- [ ] Runner гоняет продакшн-цепочку (E-3)
- [ ] Item- и run-level evaluators, включая `task_error` / `error_rate` (E-19)
- [ ] Имя рана и `run_metadata` — по E-9; run-level ключи не в metadata items (E-25)
- [ ] Версия `langfuse` запинена в lockfile (E-25)

### При первом прогоне
- [ ] Прогнаны ≥ 2 конфигурации (отличие — один параметр, E-7)
- [ ] Drill-down проверен: спаны агента вложены в trace item
- [ ] Evidence (скрин Dataset Runs) — в `summary.md` задачи
- [ ] Негативный тест: item без `reviewed_by` падает на валидации

### После прогона (каждая итерация loop)
- [ ] Протокол эксперимента заполнен: результаты, находки, **решение**; строка журнала обновлена (E-26)
- [ ] Сравнение только на одной версии датасета (E-16)
- [ ] Регрессии на зелёных items разобраны до слоя провала; при системных провалах — error analysis (К-3)
- [ ] Категории сбоев из error analysis превращены в items датасетов (К-4)
- [ ] Изменение системы оформлено новым конфигом (E-7); пороги не двигались (E-20)
- [ ] Неожиданные результаты задокументированы (rate limits, tool_choice 400, OSS-модели и tool calling)

## Правила агента (для `.cursor/rules` / skill проекта)

1. Операции контура — только повторяемыми командами (E-2); ручные правки данных в UI — не источник правды.
2. Перед выбором метрики — `metrics-guide.md`; кастомный судья без ADR запрещён (E-17).
3. Эталоны и карты — на утверждение человеку (⛔ СТОП), без апрува не продолжать (E-13).
4. Изменение системы — новый конфиг-файл; baseline не редактировать (E-7).
5. Правка датасета — новый файл-версия `vNNN_*`; существующие версии не редактировать (E-11).
6. При противоречии плана задачи и методологии — побеждает методология; противоречие подсветить человеку.
7. Эксперимент без протокола не запускать: сначала строка в журнале + протокол 🚧, по завершении — решение (E-26).
8. Онбординг создаёт только roadmap-eval и README спринта. Рабочие артефакты (dataset-map, metrics-map, конфиги, манифесты, скрипты) — только внутри задач спринта после апрува plan.md. KR roadmap — не todo текущего шага.
