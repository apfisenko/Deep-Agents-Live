# Eval-трек: что мы будем делать и зачем
## Брифинг перед практикой (читается за ~10 минут)

> Этот документ — карта местности перед стартом. Мы построим **измерительный контур качества** для нашего агента-ассистента llmstart.ru: научимся отвечать на вопрос «агент стал лучше или хуже?» цифрами, а не ощущениями. Работать будем по методологии (`docs/eval/eval-methodology.md`) — здесь же только смысл происходящего, с минимумом теории, нужной чтобы не потеряться.

---

## 1. Проблема, с которой всё начинается

У нас есть работающий агент: ReAct + RAG, отвечает по базе знаний, подбирает курсы, ведёт к оплате, фиксирует лиды, работает в вебе и Telegram. И вот вы поменяли системный промпт. Или модель — потому что новая дешевле вдвое. Или решили переехать с embedded Chroma на Qdrant.

**Стало лучше или хуже?**

С обычным кодом ответ дают тесты. С LLM так не работает: вывод вероятностный — один и тот же промпт сегодня и завтра даёт разные ответы; «правильных» ответов много; входное пространство бесконечно. Прогнать пять вопросов руками и решить «вроде норм» — это не проверка, это самообман. А цена ошибки конкретная: агент, выдумавший цену курса или несуществующую программу, — это потерянный клиент и удар по репутации.

Отсюда главный тезис всего трека:

> **Качество LLM-агента — это измеренная величина, а не впечатление.**
> Конфигурация без прогона — мнение. Прогон без зафиксированной конфигурации — шум.

## 2. AI Engineering Loop: Trace → Evaluate → Deploy

Индустрия сошлась на модели **AI Engineering Loop** ([Langfuse Academy](https://langfuse.com/academy/ai-engineering-loop)) — качество не «достигается», а поддерживается циклом:

```
Блок 1 — понять, что происходит:        Trace → Monitor (+ Error Analysis)
Блок 2 — улучшать, не ломая работающее: Datasets → Experiment → Evaluate
→ деплой → новые трейсы → цикл снова
```

Блок 1 у нас уже есть: агент инструментирован, каждый запрос пишется в Langfuse полным **трейсом** — деревом шагов (retrieval, LLM-вызовы, вызовы инструментов) с входами/выходами каждого. Трек, который мы начинаем, — это Блок 2: превратить накопленные трейсы и реальные диалоги в **датасеты**, прогонять по ним агента **экспериментами** и принимать решения по **метрикам**.

Почему трейс так важен: метрика скажет «плохо», но только трейс скажет **где** — в поиске по базе, в генерации ответа или в выборе инструмента. Всё, что мы построим, будет опираться на эту связку «цифра → конкретный шаг системы».

## 3. Что у нас на входе

- **Работающий агент** (Agent Core + MCP-инструменты: поиск по КБ, каталог, лиды, оплата) с трейсингом в Langfuse.
- **Реальные диалоги** пользователей + **отчёт анализа** этих диалогов (частоты интентов, проблемные места).
- **База знаний** `data/b2c` и `data/b2b` — из неё можно строить эталонные ответы.
- **Методология** — документ, который кодирует весь процесс: принципы, шаблоны, чек-листы, и порядок работы. Мы не будем «импровизировать eval» — мы пройдём по методологии, как по рельсам.

## 4. Датасеты: что измеряем

**Датасет** — коллекция тест-кейсов: `input` (вопрос/сценарий) → `expected_output` (что считается правильным) → `metadata`. Против него прогоняют агента при каждом изменении — вместо «задеплоили и надеемся».

Три вещи, которые важно понять до практики:

**Слоёв оценки три, и они ломаются независимо.** Retrieval (нашёл ли агент нужное в базе), генерация (правильно ли ответил по найденному), поведение (те ли инструменты вызвал, довёл ли до цели — лида). Поэтому датасет будет не один: маленькие целевые наборы по 10–30 кейсов, каждый со своей зоной ответственности. Большой датасет «на всё» — антипаттерн: когда он падает, непонятно, что чинить.

**Эталоны утверждает человек.** Это принципиально: если LLM генерирует и эталоны, и ответы, и оценку — круг замкнулся, и мы измеряем согласие модели с самой собой. Поэтому в практике будет момент, где сгенерированные черновики эталонов проходят через вас — и item без отметки `reviewed_by` физически не попадёт в систему (валидатор не пустит).

**Честность ground truth фиксируется явно.** Часть эталонов будет проверена по первоисточнику (`verified`), часть — синтезирована по базе знаний (`approximate`). Пока есть approximate, метрики годятся для **сравнения** конфигураций («B лучше A»), но не для абсолютных заявлений («качество 87%»). Это не недостаток — это честность, записанная в данные.

## 5. Метрики: чем измеряем

Правило выбора метрик в методологии называется **framework-first**, и оно лечит главную болезнь самодельных eval'ов — зоопарк кустарных «судей» со случайными промптами:

1. Если ответ проверяется кодом (правильный сегмент b2c/b2b, правильный список инструментов, лид записан) — **детерминированная проверка**. Дёшево, быстро, без шума.
2. Для стандартных потребностей — **готовые метрики** из фреймворков RAGAS и DeepEval с отлаженными промптами и документированной семантикой.
3. Свой критерий — через **G-Eval** (фреймворочный способ задать критерий словами).
4. Полностью свой судья — только через ADR с обоснованием и планом калибровки.

Стартовая пара, на которой держится минимальный контур:

| Метрика | Вопрос, на который отвечает | Зачем в паре |
|---|---|---|
| **Faithfulness** | «Все ли утверждения ответа подкреплены найденным контекстом?» — анти-галлюцинации | охраняет «агент не врёт» |
| **Task Completion** | «Решена ли задача пользователя по итогу?» — судья читает весь трейс | охраняет «агент реально помогает» |

Поодиночке каждая создаёт ложное спокойствие: можно правдиво не помогать — и можно «помогать» галлюцинациями. Это два ортогональных режима отказа ([принцип из методологии Google ADK](https://google.github.io/adk-docs/evaluate)).

И ещё одно понятие — **главная метрика**: одна north-star, по которой сравниваются конфигурации (у нас — `answer_correctness` на датасете вопросов-ответов, потому что для него есть эталоны). Остальные — guard-метрики: «конфиг лучше» значит «главная выросла, guard не просели». Без этого правила сравнение восьми метрик превращается в спор интерпретаций.

## 6. Эксперимент: сравнение конфигураций

Вопрос эксперимента — не «хорошо ли?», а «**лучше или хуже, чем было?**». Из этого следуют рабочие правила, которые вы увидите в практике:

- **Конфигурация** системы (реализация агента, модель, версия промпта, retrieval-бэкенд, версии датасетов) описывается YAML-файлом и **реально применяется** к агенту при прогоне. Каждый прогон несёт свою конфигурацию в метаданных — через полгода по нему видно, что именно гонялось.
- **Baseline неприкосновенен**, изменение = новый конфиг-файл, и между сравниваемыми конфигами — **ровно один** отличающийся параметр. Поменяли модель и промпт сразу — дельта не атрибутируется, эксперимент потерян.
- Каждый эксперимент **протоколируется**: гипотеза → конфигурация → результаты → **решение** (принят / отклонён / итерация). Журнал экспериментов отвечает на «что мы уже проверяли» — отрицательные результаты не удаляются, они защищают от повторения проверенных гипотез. Ран без протокола — измерение без вывода.

Реальный пример, зачем это всё (из чужого боевого проекта): команда прогнала модельный свип через такой контур и обнаружила, что дешёвая OSS-модель qwen3-32b обгоняет закрытую модель по точности в 1.6 раза при кратно меньшей цене. Без измерительного контура эта находка невозможна — «на глаз» такое не видно.

## 7. Как будет устроена работа: методология

Весь процесс закодирован в `docs/eval/eval-methodology.md`. Ключевая идея распределения ролей:

> **Агент занимается механикой исполнения. Человек — намерением и качеством.**

Практически это значит: coding-агент (Claude Code / Cursor) делает каркас, скрипты, черновики датасетов, прогоны и отчёты — а вы принимаете решения в ⛔ **гейтах**: утверждаете карты, эталоны, пороги, решения по экспериментам. Гейты не формальность — это ровно те места, где автоматизация без человека замыкается в самообман.

Старт — одной фразой:

> «Приступаем к eval-треку. Иди по docs/eval/eval-methodology.md, раздел "Поток работы → Подготовка": проверь предусловия, создай roadmap-eval и sprint-eval-01 из шаблонов методологии, останавливайся на каждом ⛔.»

Агент проверит предусловия, создаст roadmap трека и первый спринт из шаблонов — и остановится на ревью. Дальше — задачи.

## 8. Карта практики: спринт eval-01 «вертикальный срез»

Первый спринт намеренно узкий: не «покрыть всё», а **прошить весь путь на одном датасете** — от файла с кейсами до отчёта по baseline-прогону. Покрытие остальных датасетов — следующий спринт; срез важнее широты.

| # | Задача | Что происходит | Ваше решение (⛔) |
|---|--------|----------------|-------------------|
| 01 | Реестр конфигов + каркас | Агент учится применять `config_id`; появляется структура `evals/` и команды операций | Подтверждаете: конфиг полностью описывает систему и реально ей управляет |
| 02 | **Карта датасетов** | Из сценариев продукта, реальных диалогов и отчёта анализа выводится: какие датасеты нужны, что каждый проверяет и почему | Утверждаете карту: ЧТО измеряем |
| 03 | **Карта метрик** | Под утверждённые датасеты подбираются метрики по framework-first, фиксируются главная метрика и пороги | Утверждаете карту: ЧЕМ измеряем |
| 04 | Датасет e2e-qa | 20+ кейсов из реальных диалогов + синтетика по КБ; integrity-тесты; загрузка в Langfuse | **Утверждаете эталоны** — самый важный гейт |
| 05 | Baseline-прогон | Реальный агент (не заглушка!) прогоняется по датасету; метрики пишутся в Langfuse; протокол эксперимента | Смотрите evidence: прогон, метрики, drill-down до трейсов |
| 06 | Отчёт анализа | Топ худших кейсов с диагнозом по слоям: retrieval / генерация / поведение | Согласуете top-исправления — вход в цикл улучшения |

Обратите внимание на порядок задач 02→03: сначала «что измеряем», потом «чем» — метрики проектируются под датасеты, не наоборот. И на то, что карты — это отдельные документы с обязательным обоснованием каждого решения, а не списки из головы.

## 9. Что будет результатом

К концу практики у вас (и у проекта) будет:

1. **Работающий контур**: `make experiment` → прогон реального агента по версионированному датасету → метрики в Langfuse → отчёт.
2. **Baseline** — измеренная точка отсчёта, относительно которой теперь оценивается любое изменение.
3. **Протокол и журнал** первого эксперимента — начало истории решений проекта.
4. **Навык**, ради которого всё затевалось: спроектировать eval под *свой* агент — выбрать датасеты от сценариев, метрики от датасетов и не дать LLM оценивать саму себя.

А дальше (v0.2 трека) контур начинает окупаться: «переведи агента на Qdrant и покажи, стало ли лучше» превращается в новый YAML-конфиг, один прогон и diff двух ранов.

## 10. Глоссарий

| Термин | Смысл |
|---|---|
| **Trace / спан** | Полный след одного запроса / отдельный шаг внутри (retrieval, LLM-вызов, инструмент) |
| **Dataset / item** | Набор тест-кейсов / один кейс: input → expected_output → metadata |
| **Эксперимент (dataset run)** | Прогон агента по датасету; каждый item связан со своим трейсом |
| **Score** | Записанный результат метрики — на весь трейс или на конкретный спан |
| **LLM-as-a-judge** | Метрика, где оценку по рубрике выставляет модель (с reasoning) |
| **Reference-based / -free** | Метрика требует эталона / работает без него |
| **Baseline** | Зафиксированная конфигурация и её прогон — точка отсчёта сравнений |
| **Главная метрика / guard** | По чему решаем «лучше» / что не должно просесть при этом |
| **Eval-fix loop** | Итерации: одно изменение → прогон → сравнение → решение |
| **⛔ Гейт** | Точка, где работа останавливается до явного решения человека |
